How It Works

dbpulse is a rule-based static linter for database schemas. It works the same way ESLint catches code bugs without running your program — by analysing the structure of your DDL without connecting to a live database.

Analysis Pipeline

Your CREATE TABLE DDL
       │
       ▼
  DDL Parser (extractTableBody + parseDDL)
  ├── Balanced-paren body extraction
  ├── Column name, type, constraints extracted
  └── FK, PK, UNIQUE constraints detected
       │
       ▼
  Rule Engine (runRules — 40+ rules)
  ├── Per-table rules (R01–R27)
  ├── Cross-table rules (relationships, isolation)
  ├── Query rules (R41–R43, if queries submitted)
  └── Platform mismatch rules (per selected platform)
       │
       ▼
  Scoring Engine (scoreSecs + calcScore)
  ├── Section scores (0–100 per category)
  └── Weighted composite health score
       │
       ▼
  Schema Intelligence (generateSchemaInsight)
  ├── Single schema: domain mixing, future schema needs
  └── Multi-schema: quality gaps, duplicate tables
       │
       ▼
  Results: Issues · Summary · Solutions · Comparison · Roadmap · PDF

Quick Start

  1. Open https://db-analyzer-two.vercel.app
  2. Describe your system in Step 1 — industry, purpose, compliance needs
  3. Select your database platform (PostgreSQL, MySQL, SQL Server, Oracle, SQLite, Generic SQL)
  4. Paste CREATE TABLE statements in Step 2, or upload a .sql file
  5. Optionally add SQL queries for query anti-pattern detection
  6. Click Analyze Schema
  7. Review findings across Issues, Summary, Solutions, Comparison and Roadmap tabs
  8. Export a full PDF report

Methodology

Every rule is deterministic — it either fires or it doesn't. There is no probability score or AI inference. The DDL parser uses a balanced-parenthesis walker that correctly handles:

  • Nested function calls: DEFAULT gen_random_uuid(), DEFAULT NOW()
  • Inline REFERENCES table(id) syntax
  • Quoted identifiers: `name`, "name", [name]
  • CREATE TABLE IF NOT EXISTS
  • Standalone FOREIGN KEY ... REFERENCES and CONSTRAINT name ... syntax

Platform-Specific Rules

When a database platform is selected in Step 1, the analyser checks for type incompatibilities between your DDL and the selected platform. A warning banner appears in the Summary tab if mismatches are found. This does not affect the health score — it is advisory only.

How platform selection works Select your platform before entering DDL. The dropdown locks once DDL is entered. To change platform, clear all DDL first. Generic SQL disables platform-specific checks entirely.

PostgreSQL

The most feature-rich platform supported. Native to PostgreSQL — no warnings for these types.

✅ Native Types (no warnings)

These are valid PostgreSQL syntax
  • SERIAL, BIGSERIAL, SMALLSERIAL
  • TIMESTAMPTZ, TIMESTAMP WITH TIME ZONE
  • JSONB, JSON
  • UUID, INET, CIDR, MACADDR
  • TSVECTOR, TSQUERY (full-text)
  • HSTORE, ARRAY types
  • INT4RANGE, INT8RANGE

⚠️ Incompatible Types (warning shown)

These belong to other platforms
  • AUTO_INCREMENT → use SERIAL
  • NVARCHAR → use VARCHAR
  • DATETIME → use TIMESTAMPTZ
  • NUMBER → use NUMERIC/DECIMAL
  • VARCHAR2 → use VARCHAR
  • TINYINT → use SMALLINT
  • LONGTEXT / MEDIUMTEXT → use TEXT

MySQL / MariaDB

✅ Native Types (no warnings)

  • AUTO_INCREMENT
  • TINYINT, MEDIUMINT
  • LONGTEXT, MEDIUMTEXT, TINYTEXT
  • ENUM, SET
  • YEAR
  • JSON (MySQL 5.7+)

⚠️ Incompatible Types (warning shown)

  • SERIAL → use AUTO_INCREMENT
  • BIGSERIAL → use BIGINT AUTO_INCREMENT
  • TIMESTAMPTZ → use DATETIME or TIMESTAMP
  • JSONB → use JSON
  • INET → use VARCHAR(45)
  • BOOLEAN → use TINYINT(1)
  • NVARCHAR → use VARCHAR with utf8mb4

Microsoft SQL Server

✅ Native Types (no warnings)

  • NVARCHAR, NCHAR, NTEXT
  • DATETIME2, DATETIMEOFFSET
  • UNIQUEIDENTIFIER
  • MONEY, SMALLMONEY
  • BIT
  • IMAGE, ROWVERSION

⚠️ Incompatible Types (warning shown)

  • SERIAL → use IDENTITY(1,1)
  • BIGSERIAL → use BIGINT IDENTITY
  • AUTO_INCREMENT → use IDENTITY
  • TIMESTAMPTZ → use DATETIMEOFFSET
  • JSONB → use NVARCHAR(MAX) + JSON functions
  • INET → use VARCHAR(45)
  • BOOLEAN → use BIT

Oracle Database

✅ Native Types (no warnings)

  • VARCHAR2, NVARCHAR2
  • NUMBER (replaces INT, DECIMAL, FLOAT)
  • CLOB, NCLOB, BLOB
  • RAW, ROWID, UROWID
  • DATE (includes time in Oracle)

⚠️ Incompatible Types (warning shown)

  • SERIAL → use SEQUENCE + TRIGGER
  • AUTO_INCREMENT → use GENERATED ALWAYS AS IDENTITY
  • BOOLEAN → use NUMBER(1) CHECK (col IN (0,1))
  • TIMESTAMPTZ → use TIMESTAMP WITH TIME ZONE
  • JSONB → use CLOB with IS JSON constraint
  • TINYINT → use NUMBER(3)
  • NVARCHAR → use NVARCHAR2

SQLite

SQLite Type Affinity SQLite uses dynamic typing with 5 storage classes: NULL, INTEGER, REAL, TEXT, BLOB. Most type names are accepted but stored using affinity rules. The warnings below flag types that have no SQLite equivalent.

✅ Native Types (no warnings)

  • INTEGER (auto-increment via ROWID)
  • REAL
  • TEXT
  • BLOB
  • NUMERIC

⚠️ Incompatible Types (warning shown)

  • SERIAL → use INTEGER PRIMARY KEY
  • BIGSERIAL → use INTEGER PRIMARY KEY
  • TIMESTAMPTZ → use TEXT (ISO 8601) or INTEGER (Unix epoch)
  • JSONB → use TEXT
  • INET → use TEXT
  • UUID → use TEXT or BLOB
  • BOOLEAN → use INTEGER (0/1)

Generic SQL

No platform warnings Selecting Generic SQL disables all platform-specific type checks. Use this when your DDL is written for multiple platforms or when the target platform is not yet decided. All 40+ schema rules still apply.

Health Score Formula

Step 1 — Section Score

Each of the 6 scored sections starts at 100 and deducts penalty points per finding:

Section Score = max(0, 100 − Σ(penalty × domain_multiplier))

Step 2 — Weighted Composite Score

Health Score =
  (Keys & Integrity    × 0.25) +
  (Normalization       × 0.20) +
  (Schema & Arch       × 0.20) +
  (Security            × 0.15) +
  (Indexing & Perf     × 0.10) +
  (Relationships       × 0.10)

FAQ

Is my DDL sent to a server?

No. All analysis runs entirely in your browser using JavaScript. No DDL, query text, or schema data is transmitted to any server at any point. The tool works fully offline after the first page load.

How accurate are the findings?

Every rule is deterministic — it either fires or it doesn't. There is no AI inference or probability scoring. However, some rules rely on naming conventions (e.g. detecting PII from column names like ssn). If your column names are non-standard, some findings may not fire. Conversely, if you use generic names for non-PII columns (e.g. a column named ssn that stores something else), you may see false positives.

Why doesn't the platform dropdown affect my score?

Platform mismatch warnings are advisory only — they don't affect the health score. This is intentional: a schema that uses PostgreSQL types is not inherently worse than one using MySQL types. The warning tells you there's a mismatch between your DDL and selected platform so you can investigate, but doesn't penalise you for it.

Can I analyse multiple schemas?

Yes — up to 5 schemas can be submitted simultaneously. The Comparison tab shows a side-by-side health score breakdown. All schemas share the same platform selection.

What SQL dialects are supported?

The parser is designed for ANSI SQL with support for PostgreSQL, MySQL, SQL Server, Oracle, and SQLite syntax. CREATE TABLE IF NOT EXISTS, inline REFERENCES, CONSTRAINT name syntax, and function calls in DEFAULT values are all handled correctly.

How do I improve my score?

Fix Critical findings first — each Critical costs 15 points from its section. Focus on high-weight sections: Keys & Integrity (25%) and Normalization (20%) have the most impact on your overall score. The Solutions tab provides a prioritised fix list with ready-to-run SQL.
